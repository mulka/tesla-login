Passkeys

A modern authentication approach for the Tesla browser

What Are Passkeys?

Passkeys are a passwordless authentication standard built on the WebAuthn (Web Authentication) API. Instead of typing a password, your device proves your identity using public-key cryptography — the same math that secures HTTPS connections.

When you create a passkey, your device generates a unique key pair:

  • Private key — stays on your device, never leaves, never shared
  • Public key — sent to the website and stored on their server

To log in, the server sends a challenge. Your device signs it with the private key. The server verifies the signature with your public key. No password ever crosses the network.

💡 Passkeys are phishing-resistant by design. They're cryptographically bound to the specific website domain, so they can't be tricked into working on a fake lookalike site.

  1. Registration: You visit a site and choose to create a passkey. Your device creates a key pair and sends the public key to the server.
  2. Authentication: When you return, the server sends a random challenge. Your device signs it with the private key (after biometric or PIN confirmation).
  3. Verification: The server checks the signature against your stored public key. If it matches, you're in.

The Tesla Browser Challenge

The Tesla in-car browser is a Chromium-based browser, but it runs in a unique environment with some key constraints:

⚠️ No built-in passkey storage: The Tesla browser doesn't have a system-level credential manager like Android, iOS, or desktop Chrome. There's no Face ID, fingerprint sensor, or TPM chip in the car.

⚠️ No browser extensions: You can't install a password manager or authenticator extension in the Tesla browser.

⚠️ Shared device: The car's browser may be used by multiple drivers. Traditional device-bound credentials don't map cleanly to this model.

Cross-Device Authentication: QR Code Flow

This is where the QR code login approach comes in. WebAuthn supports a cross-device authentication flow (sometimes called hybrid transport or caBLE) that works like this:

✅ The Tesla browser displays a QR code. Your phone — which does have passkey support — scans it, authenticates you via biometrics, and securely relays the credential back to the browser session.

  1. The web app on the Tesla screen initiates a WebAuthn authentication request
  2. Since no local authenticator is available, it displays a QR code containing a one-time pairing payload
  3. The user scans the QR code with their phone (which has passkeys synced via iCloud Keychain, Google Password Manager, etc.)
  4. The phone and Tesla browser establish a secure Bluetooth or network-proxied channel
  5. The phone performs the passkey authentication (Face ID, fingerprint, etc.)
  6. The signed assertion is relayed back to the Tesla browser
  7. The server verifies it — user is logged in on the Tesla screen

Why This Matters for Tesla Web Apps

  • No typing passwords on the Tesla touchscreen — the on-screen keyboard is clunky, especially while parked
  • Phishing resistant — even if someone puts a fake login page on the Tesla browser, passkeys won't work on the wrong domain
  • Per-driver identity — each driver authenticates with their own phone, enabling personalized app experiences
  • Session management — the web app can tie sessions to driver profiles and expire them on vehicle logout
  • No shared passwords — eliminates the "saved password in the car browser" problem

Implementation Considerations

Building this for the Tesla browser requires checking a few things:

  • WebAuthn API support: Does the Tesla Chromium build expose navigator.credentials? If so, the standard cross-device flow may work natively.
  • Bluetooth availability: The standard caBLE protocol uses BLE for proximity verification. If Tesla's browser sandbox blocks Bluetooth, a network-relay fallback is needed.
  • Custom QR flow: If native WebAuthn cross-device isn't available, the app can implement its own QR-based authentication — the Tesla screen shows a code, the phone hits an API endpoint, and the browser polls or uses WebSockets to detect the completed auth.
  • Session tokens: Once authenticated, store a session token in the browser's localStorage or a cookie to avoid re-scanning every page load.

💡 Even if the Tesla browser doesn't support the full WebAuthn hybrid flow natively, a custom implementation using QR codes + phone-based authentication achieves the same user experience — scan, approve, done.
